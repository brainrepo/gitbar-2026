## Descrizione

Registrato in macchina con le cuffiette (grazie ai consigli di Alex Racuglia), abbiamo esplorato le Progressive Web App, le "Superman web app" che Google ha lanciato nel 2015. Abbiamo raccontato come Jobs nel 2007 voleva che l'iPhone usasse web app, poi ha cambiato idea in 4 mesi lanciando l'App Store (e facendo 40 milioni di revenue), e come oggi le PWA cercano di prendere il meglio dei due mondi: linkabili e trovabili su Google come le web app, ma con notifiche push e funzionamento offline come le app native.

## Takeaway

- Le PWA con i Service Workers possono gestire la connessione intermittente (il vero problema, non l'offline totale) usando cache controllata via JavaScript diversa dalla cache del browser
- Il web manifest è un file JSON che dice al device come "vestire i panni" di un'app nativa: nome, icona, colore della top bar
- Con PurgeCSS si riducono drasticamente le dimensioni eliminando CSS inutilizzato, ma le PWA consumano comunque più energia e dati delle native perché devono scaricare anche lo skeleton dell'app
- Le notifiche Web Push funzionano su Android ma non su iOS: Apple non le attiverà mai perché intaccherebbe il vantaggio competitivo delle app native (e lo store)
- Gli sviluppatori HTML/CSS/JavaScript costano meno degli sviluppatori Objective-C/Swift/Java, rendendo le PWA ideali con budget limitato

## Bold Opinion

- L'assenza di uno store per le PWA è un vantaggio, non un limite: puoi pubblicare subito senza code di approvazione e senza dare percentuali a nessuno
- iOS limita deliberatamente le PWA (no Web Push, supporto offline parziale) per proteggere l'App Store: è strategia di business, non limitazione tecnica
- Per app multiplataforma le PWA battono le native: sviluppi una volta sola, aggiorni con un deploy, nessun tempo di compilazione o autorizzazione store
